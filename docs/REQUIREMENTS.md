# 要件定義

## プロダクト概要

「複利活用 習慣化カウンター」は、習慣化したいタスクの実行時間をストップウォッチで記録し、昨日の自分を少しずつ超えるためのアプリです。

毎日の実績をカレンダーに表示します。利用者が今日の目標時間を設定すると、昨日の実績から何パーセント増減した目標なのかを自動表示し、努力が積み上がっている感覚を得られるようにします。

## ユーザー

- Google アカウントでログインする利用者
- 複数ユーザーが同じアプリを利用する
- 各ユーザーのデータは完全に分離する

## 基本機能

### 認証

- Google 認証でログインできる。
- ログアウトできる。
- 未ログイン時は保護ページにアクセスできない。

### タスク管理

- 習慣化したいタスクを作成できる。
- タスク名を変更できる。
- タスクごとに今日の目標時間を設定できる。
- タスクをアーカイブできる。
- 自分のタスクだけを表示・操作できる。

### ストップウォッチ計測

- タスクごとにストップウォッチを開始できる。
- 計測中の経過時間を表示できる。
- 計測を停止して履歴として保存できる。
- 保存された計測履歴は削除せず保持する。
- 計測履歴から日別の合計時間を算出できる。

### カレンダー表示

- タスクごとに日別の実績時間をカレンダーに表示できる。
- 今日、昨日、過去日の実績を確認できる。
- 実績がない日は 0 として扱う。

### 昨日の自分超え

- 利用者が今日の目標時間を設定できる。
- 今日の目標時間と昨日の実績時間から、前日比を自動計算する。
- 今日の実績が目標に対してどこまで進んでいるか表示する。
- 昨日の実績を超えたことが分かる表示を行う。
- 今日の目標を達成した後は、前日比よりも目標達成を優先して表示する。
- 昨日の実績が 0 秒の場合は、前日比を表示しない。
- 今日の目標時間と実績時間を日別履歴として保持する。

前日比の計算:

```text
前日比 = (今日の目標時間 - 昨日の実績時間) / 昨日の実績時間 * 100
```

表示例:

```text
昨日の実績: 20分
今日の目標: 22分
昨日より +10%
```

### 継続日数（ストリーク）と履歴

- **継続日数の定義**:
  - 利用者のタイムゾーン（デフォルト `Asia/Tokyo`）を基準とする。
  - 1日のうち、何かしらの習慣タスクに対して合計1秒以上の取り組み（計測完了したセッションがある、または現在計測中のセッションがある）が存在する場合、その日は「取り組み日」としてカウントする。
  - 今日（判定を行う時点のローカル日付）または昨日から連続して遡れる「連続取り組み日数」を「現在継続日数」とする。
  - これまでの取り組み履歴の中で最も連続した日数を「最長継続日数」とする。
- **継続履歴ページの表示**:
  - 現在の継続日数（ストリーク数）と過去の最長継続日数を分かりやすく表示する。
  - 過去数ヶ月分の活動状況をカレンダー（またはヒートマップ）形式で表示し、取り組んだ日を視覚的にハイライトする。
  - 過去の日別の詳細（どのタスクを何分取り組んだか）の履歴リストを表示する。

## 非機能要件

- Docker Compose でアプリと DB を起動できる。
- Oracle Cloud Infrastructure へのデプロイを想定する。
- Node.js runtime で動作する。
- npm で依存関係を管理する。
- PostgreSQL を使用する。
- DB migration はアプリ起動時に `prisma migrate deploy` で適用する。
- TypeScript の `any` を禁止する。
- `.ts` ファイルの unit test coverage は 100% とする。
- domain / application のロジックは TDD で、期待する振る舞いを先に unit test として定義する。
- coverage 100% は、到達不能、未使用、冗長なロジックや不要な分岐を残さないための制約として扱う。
- coverage の数値を満たすためだけの無意味なテストや、安易な対象除外を行わない。
- `.tsx` の画面挙動は Playwright でテストする。
- repository、migration、DB 制約は PostgreSQL を使った integration test で検証する。
- ドキュメントとコメントは日本語にする。
- Husky で Git hooks を管理する。
- lint-staged で staged file を検査する。
- commitlint で Conventional Commits を検査する。
- lint、型検査、unit test、coverage の必須チェックを通過しない場合は push できないようにする。
- `any`、`as any`、`@ts-ignore`、`@ts-nocheck` を禁止する。
- import 境界違反と循環依存を静的解析で禁止する。
- CI で format、lint、typecheck、unit test、coverage、integration test、Playwright E2E、production build、Docker build、migration、依存関係脆弱性を検証する。
- CI の必須チェックを通過しない変更は main branch に merge できないようにする。

## 初期スコープ

初期リリースで含める:

- Google 認証
- タスク作成
- タスク一覧
- ストップウォッチ開始・停止
- 計測履歴保存
- タスク別カレンダー
- 今日の目標時間設定
- 昨日の実績に対する前日比表示
- multi-user 対応
- Docker Compose 起動

初期リリースでは含めない候補:

- 通知
- PWA
- データ export / import
- 複数 provider 認証
- チーム共有
- 高度な分析画面

## 重要な仕様メモ

- 「今日」「昨日」はユーザーのタイムゾーンで判定する。
- DB には UTC で保存する。
- デフォルトタイムゾーンは `Asia/Tokyo` とする。
- 計測履歴は削除せず、集計の元データとして保持する。
- 削除操作が必要な場合も、物理削除ではなく archive / soft delete を優先する。
- 同時に計測できるタスクは、ユーザーにつき 1 件とする。
- 計測開始時刻を DB に保存し、ブラウザを閉じても計測を継続する。
- 日付をまたぐ計測は、ユーザーのタイムゾーンを基準に日別分割する。
- 複数端末で操作された場合は、DB 上の計測状態を正とする。

## UI・UX 方針

- スマートフォンを優先して設計する。
- シンプルで洗練された、落ち着いた道具感を重視する。
- タスク名、今日の実績、今日の目標、計測開始・停止を主要情報とする。
- 主要操作までの手順と画面内の情報量を抑える。
- 過剰なグラデーション、発光、紙吹雪、派手なアニメーションを使用しない。
- 達成状態は、色、短い文言、控えめな動きで表現する。
- 安っぽいカードの多用や、大きすぎる角丸を避ける。
- カレンダーは月表示を基本とし、各日の実績時間を読み取れるようにする。
